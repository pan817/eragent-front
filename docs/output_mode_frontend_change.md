# `output_mode` 改造 — 前端必做协调说明

> 发起方：后端
> 发布日期：2026-04-16
> 配套文档：[sse_react_backend.md](./sse_react_backend.md) / [prompt_issue.md](./prompt_issue.md)
> 状态：**前端必须改 1 行**（详见 §3）

---

## 1. 背景

### 1.1 问题观察

联调期间发现，**所有 ReAct 路径的回复都被强行套上"## 报告"标题 + 摘要/建议四段结构**，包括明显是事实查询的场景，例如：

- "查最新的一个 PO" → 回复以 "## 采购订单查询结果\n\n### 摘要\n..." 开头
- "PO-2024-0001 在哪一步" → 同样四段报告格式

事实查询场景使用报告格式让用户感到"过度形式化"，体验异常笨重。

### 1.2 根因（后端侧）

后端原 P2PAgent system prompt 强制要求"以 Markdown 格式组织报告，包含标题、摘要、详细发现和建议"，对**所有意图**无差别生效。后端 P0 改造已按查询性质分档 prompt（事实查询直答 / 分析查询出报告）。

但仅改 P0 不够：`output_mode` 字段的 Pydantic 默认值是 `"detailed"`，后端**无法区分**两种场景：

| 来源 | 后端看到的值 | 用户真实意图 |
|---|---|---|
| 前端**不传** `output_mode` | `"detailed"`（schema 默认） | "我不知道，由系统决定" |
| 前端**显式传** `"detailed"` | `"detailed"` | "我就要详细报告" |

为消除这个歧义，本次改造引入 `auto` 模式作为新默认值。

## 2. 后端契约变更（已上线/向后兼容）

### 2.1 `output_mode` 枚举

| 值 | 旧 | 新 | 说明 |
|---|---|---|---|
| `auto` | ❌ 不存在 | ✅ **新默认** | 后端按查询性质自动决策（事实查询 → chat，分析/合规等 → detailed） |
| `detailed` | ✅ 默认 | ✅ 显式可选 | Markdown 报告 4 段结构，**显式传时严格被尊重**，后端不覆盖 |
| `brief` | ✅ 显式可选 | ✅ 不变 | 简报摘要 3-5 要点 |
| `table` | ✅ 显式可选 | ✅ 不变 | 表格 + 简短结论 |
| `chat` | ❌ 不存在 | ✅ 显式可选 | 自然对话直答，无标题/建议段落 |

### 2.2 默认值变化

```diff
- output_mode: str = Field(default="detailed", pattern="^(detailed|brief|table)$")
+ output_mode: str = Field(default="auto", pattern="^(auto|detailed|brief|table|chat)$")
```

### 2.3 行为契约

| 前端传值 | 后端行为 |
|---|---|
| 不传 / 传 `auto` | 按 `intent_kind` 自动决策：`data_lookup → chat`；其余 → `detailed` |
| 显式 `detailed/brief/table/chat` | **严格使用，不覆盖**（`【格式强制覆盖】` 锚定，能压住 system prompt 的"按查询性质"判断） |

### 2.4 DAG 路径降级保护（透明）

DAG 路径下的 ReportAgent 主体定位是"报告生成器"，与 `chat` 模式语义冲突。后端做透明降级：

- DAG 路径 + `effective_output_mode == "chat"` → 自动升级为 `brief`
- 前端**无需感知**，行为正确即可

### 2.5 向后兼容性

| 旧前端调用 | 后端响应 | 是否报错 |
|---|---|---|
| 不传 `output_mode` | 走 auto 自动决策（**新行为，体验更智能**） | ❌ 不报错 |
| 显式传 `"detailed"` | 严格按 detailed 输出（行为同旧版） | ❌ 不报错 |
| 显式传 `"brief"` / `"table"` | 同上 | ❌ 不报错 |
| 传未知值（如 `"freeform"`） | 422 Unprocessable Entity | ✅ Pydantic 校验 |

**接口层面 100% 向后兼容，不会让任何旧前端调用失败。**

## 3. 前端**必须**改动（1 行 + UI 同步）

### 3.1 必改：默认请求体

> ⚠️ 前端**当前若硬编码 `output_mode: "detailed"`**（联调期间已确认），必须修改为以下两种之一。

#### 选项 A（推荐）：删除字段

```diff
{
  "query": "...",
  "user_id": "...",
- "output_mode": "detailed",
  "auto_persist": true
}
```

后端默认值就是 `auto`，省略字段等价于 `auto`。

#### 选项 B：显式传 `"auto"`

```diff
{
  "query": "...",
  "user_id": "...",
- "output_mode": "detailed",
+ "output_mode": "auto",
  "auto_persist": true
}
```

更直观，Code Review 时一眼看出意图。

### 3.2 必改：UI 选项

> ⚠️ 当前前端 UI 没有 `auto` 选项。**必须**在选择控件（如下拉框 / 按钮组）里加上"自动（推荐）"作为新默认值。

#### UI 改造示意

| 旧 UI | 新 UI |
|---|---|
| 默认选中"详细" | **默认选中"自动（推荐）"** |
| 选项：详细 / 简报 / 表格 | 选项：**自动**（默认）/ 详细 / 简报 / 表格 / 对话 |

#### 各选项语义提示（建议加 tooltip）

| UI 文案 | 值 | tooltip |
|---|---|---|
| 自动（推荐） | `auto` | 系统按查询性质自动决定格式：事实查询用对话风格，分析查询用详细报告 |
| 详细报告 | `detailed` | 强制 Markdown 报告四段结构（标题+摘要+发现+建议） |
| 简报 | `brief` | 强制 3-5 个要点形式，控制在 500 字 |
| 表格 | `table` | 优先用 Markdown 表格 + 简短结论 |
| 对话 | `chat` | 强制自然对话直答，不加标题与建议段落 |

### 3.3 不改的后果（明确告知）

如果前端不改：

| 场景 | 实际行为 | 用户感知 |
|---|---|---|
| 前端硬编码 `detailed`，用户问事实查询 | 仍输出 detailed 报告（标题+摘要+建议） | 体验仍然笨重，**P0+P1 改进白做** |
| 前端 UI 默认 detailed，用户没改下拉 | 同上 | 同上 |

**前端不改 = 后端这次改造对终端用户零感知。**

## 4. 兼容性矩阵（过渡期）

后端先上线、前端尚未改的过渡期内的真实行为：

| 后端版本 | 前端版本 | 用户事实查询体验 | 备注 |
|---|---|---|---|
| 改造前 | 改造前 | 笨重报告 | 旧行为 |
| **改造后** | **改造前** | **仍然笨重** | 后端尊重前端显式 detailed |
| 改造后 | 改造后（推荐方案 A 或 B）| 自然对话 | 全链路改造完成 |
| 改造前 | 改造后 | 触发 422 | 因为旧后端 pattern 不识别 `auto` |

**关键约束**：后端先于前端上线安全；前端先于后端上线会触发 422，**禁止前端先发**。
